## 6. The Production 4-Pillar Relational Schema (LOCAL_PRISMA Specifications)

When `ACTIVE_MODE = "LOCAL_PRISMA"`, the local PostgreSQL engine enforces rigid data integrity. All schemas must prevent data corruption through strict cascade bindings and precise type choices.

### Schema Enforcement Blueprint:
* **Pillar 1 - Categories:** Must act as the foundational taxonomy. Deleting a category scope must trigger a cascaded wipe across all connected assets.
* **Pillar 2 - FAQs:** Knowledge base modules managed via explicit integer ordering (`sortOrder`) and an instantaneous activation state (`isActive`).
* **Pillar 3 - Site Settings:** Monolithic dictionary mapping application states to unique key strings.
* **Pillar 4 - Items & Image Architecture:** Items never store flat image URL strings. They map onto a separate relation table (`CollectionItemImage`) enabling standalone alt tags, ordering, and a distinct cover asset toggle (`isCover: Boolean`). The `price` column must remain an optional Float (`Float?`) to maximize product-agnostic collection capabilities.

## 7. The Security Doorman Layer & Route Guards

The administrative boundary is binary and protected at the routing edge. 

### Middleware Rules for AI Agents:
* **Double-Gated Isolation:** 1. Private Admin Page Scopes (`/admin/:path*`) must immediately trigger a secure server-side redirect to `/login` if no active session is found.
  2. Admin API Route Scopes (`/api/admin/:path*`) must NEVER issue a redirect. They must abort instantly, returning a clean `401 Unauthorized` JSON payload. This guarantees that frontend asynchronous data fetches fail gracefully instead of breaking the client-side execution loop.
* **Session Management:** Authentication states are forged inside `src/app/api/auth/login/route.ts` against the server environment variable `process.env.ADMIN_PASSWORD`. A successful token match injects an `HttpOnly`, `Secure` (in production), `SameSite=Lax` cookie named `admin_session` directly into the header. Client-side mutations of this cookie are strictly prohibited.

## 8. Data Ingestion & Transaction Security

All server-side endpoints must operate defensively against corrupt mutations.

### Execution Mandates:
* **Strict Payload Parsing:** Direct payload ingestion from form structures into the database is illegal. Numeric input string data (e.g., prices) must be cast explicitly using numbers parsing logic (such as Zod preprocessors using `parseFloat()`) before hitting Prisma. This annihilates runtime `NaN` errors and database-level `400 Bad Requests`.
* **Transactional Bulk Modifications:** Multiple configuration fields (such as Site Settings arrays) must be committed atomically using Prisma transactions (`prisma.$transaction`). The engine must execute an `.upsert()` operation for every single key parameter, ensuring changes either apply completely as a single unit or abort entirely on database friction.

## 9. Local Binary Storage Pipeline

To prevent vendor lock-in and keep local environments lightweight, the media engine writes files directly to the server asset grid.

### File Ingestion Rules:
* **Multi-Part Form Extraction:** Media endpoints (`/api/admin/upload`) must intercept raw `FormData`, unpack binary file streams, and strictly validate standard MIME types (restricting uploads to stable web imagery: JPEG, PNG, WEBP).
* **Recursive Storage Creation:** Files must write directly into the local repository target: `public/uploads/`. The script must verify and create missing physical folders programmatically using asynchronous recursive directory creation (`fs.mkdir` with `{ recursive: true }`).
* **Filename Hashing Standard:** Original filenames must be discarded on arrival to eliminate encoding collisions. Every file must receive a crytographically secure random UUID token (`crypto.randomUUID()`) joined cleanly with its native extension.

## 10. High-Contrast Monochrome Design & Accessibility Specification

This monolith adheres to a strict **High-Contrast Monochrome Tech** design system, striking an elite balance between minimal luxury aesthetics (Linear/Vercel inspired) and strict barrierefrei accessibility (WCAG compliant).

### Design Rules and Visual Restrictions:
* **The Absolute `transition-all` Ban:** You are completely forbidden from applying the generic `transition-all` class to interactive components. It causes sub-pixel text rendering blur, performance spikes, and layout flickering during active state transitions. You must write explicit target properties (`transition-[color,background-color,border-color,transform]`) constrained to a lightning-snappy `duration-[100ms]` animation curve.
* **Haptic Tap Curves:** Interactive elements (buttons, inputs, sliders) must deliver tangible physical feedback via immediate downscaling upon cursor click (`active:scale-[0.97]` or `active:scale-[0.99]`).
* **Defensive Fallback Styling:** To permanently insulate the design system from thematic color drift during automated refactorings, the core global utility variable `--color-brand-500` inside `globals.css` must be mapped directly to our contrast monochrome accent. Any third-party asset or component defaulting to Tailwind's active state will automatically conform to our monochrome rules instead of leaking unauthorized blues, purples, or generic grays.
* **Light Mode Elevation Law:** Light mode must never render as flat, blinding white. The base canvas must utilize a comfortable, soft matte off-white (`#f5f5f7`). All layout containers, tables, and interaction fields must sit elevated as solid white blocks (`bg-white`) casting an ultra-soft, diffused drop shadow (`shadow-sm` or equivalent low-opacity boundaries) with heavy, visible outlines (`border-neutral-300`). Primary text colors must resolve strictly to high-contrast black (`text-black` / `#000000`).

## 11. Lightweight Non-Dependency i18n Architecture

Multilingual rendering is deployed natively on the client side using static JSON dictionaries without bloating the JavaScript runtime with heavy third-party translation packages.

### Dictionary Guidelines:
* **Taxonomy & Paths:** Standard configuration definitions reside under `src/dictionaries/` as `en.json` and `de.json`.
* **Recursive Path Resolution:** Translations are distributed through a optimized React Context and Custom Hook (`useTranslation()`). The `t(key)` utility must split string tokens dynamically (e.g., `'sidebar.navigation.dashboard'`) to traverse nested JSON structures recursively.
* **Safe State Fallbacks:** Initial language configuration reads from `localStorage` (defaulting to English `'en'`). If an active string parameter is completely missing inside the active language dictionary, the resolution engine must gracefully fall back onto the identical English dictionary object path rather than displaying empty text blocks or breaking the UI render tree.
